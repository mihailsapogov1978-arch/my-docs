    Личный кабинет и ИИ

    РА по настройке Модуля для выделения ключевой информации из отчетных документов¶

    Сервис поставляется с Docker Compose файлом для удобной настройки и запуска.

    Требования

    Перед запуском сервиса убедитесь, что у вас установлен Docker и Docker Compose на сервере. Эти инструменты необходимы для контейнеризации и управления многоконтейнерными приложениями.

    Запуск¶

    1. Распакуйте архив с дистрибутивом.
    2. Перейдите в каталог с ним.
    3. Запустите сервис с помощью команды: docker compose up -d. Это поднимет все необходимые контейнеры (extraction-service и llama-cpp).

    Остановка сервиса¶

    Для остановки всех контейнеров выполните: docker compose down

    Перезапуск сервиса¶

    Для перезапуска контейнеров выполните: docker compose down && docker compose up -d Эта команда сначала остановит контейнеры, а затем запустит их заново.

    Переменные окружения¶

    Для правильной работы сервиса необходимо настроить следующие переменные окружения в docker-compose.yml:

    LLM.URL: URL для обращения к модели. Пример: http://llama-cpp:6666.

    LLM.API_KEY: API ключ для доступа к модели. Пример: ollama.

    LLM.MODEL: Идентификатор модели для использования. Пример: hf.co/Qwen/Qwen2.5-3B-Instruct-GGUF.

    Группа параметров для определения выбора алгоритма парсинга сканов билетов (по умолчанию: llm):¶

    SCAN_PARSER_AERO: Параметры парсера для сканов билетов Aeroflot. Пример: classic.

    SCAN_PARSER_RZD: Параметры парсера для сканов билетов РЖД. Пример: classic.

    SCAN_PARSER_YAMAL1: Параметры парсера для сканов билетов Ямал 1. Пример: classic.

    SCAN_PARSER_YAMAL1_NEW: Параметры парсера для сканов билетов Ямал 1 нового вида. Пример: classic.

    SCAN_PARSER_YAMAL2: Параметры парсера для сканов билетов Ямал 2. Пример: classic.

    SCAN_PARSER_KVIT_YAMAL: Параметры парсера для квитанций Ямал. Пример: classic.

    SCAN_PARSER_S7: Параметры парсера для сканов билетов S7. Пример: classic.

    SCAN_PARSER_POBEDA: Параметры парсера для сканов билетов Победа. Пример: classic.

    SCAN_PARSER_URAL: Параметры парсера для сканов билетов Урал. Пример: classic.

    Структура контейнеров¶

    extraction-service: Основной сервис для обработки данных. Обрабатывает сканы билетов и использует модели для генерации результатов.

    llama-cpp: Сервер инференса для больших языковых моделей, который обрабатывает запросы на основе выбранной модели. Слушает на порту 6666

    Примечания

    Все необходимые модели и парсеры должны быть размещены в директориях, указанных в docker-compose.yml:

    ./llm-parsers — парсеры для моделей.
    ./models — LLM модели формата GGUF.

    Tip

    Если у вас возникнут вопросы или проблемы с настройкой, пожалуйста, обратитесь к документации Docker или свяжитесь с разработчиками.

    ЧТЗ на ИИ¶

    Общие сведения

    Модуль предназначен для обработки с применением алгоритмов искусственного интеллекта отчетных документов в цифровой форме, в том числе в виде изображений, с целью выделения существенной значимой информации из этих документов (распознавание) и передачи этой информации в структурированном виде в ВИС.

    Модуль включает в себя ИИ-компонент распознавания сущностей и дополнительные настройки под индивидуальные потребности Заказчика согласно требованиям Контракта.

    Функциональные возможности ИИ-компонента распознавания сущностей:¶

    • преобразование фотографических изображений документов в текстовые;
    • выявление и представление в структурированном машиночитаемом виде именованных сущностей, содержащихся в документах;
    • классифицирование документов по заранее заданным шаблонам;
    • возможность определения и обработки различных типов документов в многостраничных файлах формата pdf;
    • возможность обобщения и преобразования сущностей, найденных в документах, в соответствии с заданными настройками, используя большие языковые модели;
    • возможность протоколирования действий.

    Для возможности выполнения Модулем своих функций распознавания конкретных видов и форм отчетных документов Модуль должен иметь возможность настройки отчетных документов.

    Согласно требованиям ТЗ Контракта в результате настройки Модуль должен выполнять следующие функции: - классифицировать виды отчётных документов, закрепленные в пункте 3.1.2 ТЗ Контракта: маршрут-квитанция, электронный билет, билет, счета, квитанции, акты, товарные и кассовые чеки за проживание, контрольные купоны, накладные.

    Работу по подготовке экземпляров отчетных документов, необходимых для обучения Модуля, Исполнитель должен проводить на основе образцов документов, предоставленных Заказчиком;

    1. распознавать данные из отчетных документов в объеме, необходимом для формирования записей в карточке электронного документа на вкладке «Расходы» страницы «Отчет о расходах» веб-приложения «Сервис подотчётного лица» ГИС Смета.
      Для отчетных документов вида «билет» перечень компаний-перевозчиков и перечень распознаваемых данных должен соответствовать пунктам 2.6.2.1, 2.6.2.2 ТЗ Контракта;

    2. распознавать изображения отчетных документов, представленных в форматах pdf, png, jpg, jpeg, tiff, при выполнении следующих условий: изображение содержит всю необходимую информацию;

      • все символы изображения читаемы;
      • разрешение изображения не менее 200dpi;
      • изображения повернуто не более чем на 30 градусов;
      • предоставлять сервис взаимодействия с ВИС, обеспечивающий получение из ВИС отчетных документов в виде файлов формата pdf, png, jpg, jpeg, tiff и возвращение в ВИС данных, полученных из отчетных документов после распознавания, в машиночитаемом виде.

    Требования к информационному взаимодействию Модуля с ВИС¶

    Для получения доступа к данным Модуля требуются токен-авторизации. Токен-авторизация должна использоваться для каждой операции-запроса. Модуль может иметь множество токенов-авторизации для разных получателей данных. Срок жизни токен-авторизации должен задаваться администратором. Выдача токенов-авторизации должна производиться через службу технической поддержки на портале самообслуживания Исполнителя. REST-сервис Модуля содержит одну значимую функцию parse_all, которая принимает файл из ВИС и возвращает ВИС результат распознавания в JSON формате. Передаваемый из ВИС в Модуль файл может быть в формате pdf, который может содержат как текст, так и результат сканирования, т.е. изображение. Так же из ВИС в Модуль могут передаваться файлы форматов png, jpg, jpeg, tiff с изображением.

    Описание возвращаемого Модулем результата распознавания в JSON формате¶

    Модуль возвращает коллекцию билетов в формате JSON. Каждый элемент коллекции представляет один или группу билетов, объединённых одной суммой.

        "need_to_check": false
      }

    Поле Тип Описание
    price string Сумма за билет или билеты, если в документе присутствует только одна сумма по всем билетам
    tickets array Набор билетов, объединённых одной суммой
    Элемент билета
    name string ФИО пассажира
    company string Компания-перевозчик
    id string Номер билета
    purchase_day string Дата покупки
    dates array Даты отлёта и прилёта
    places array Место отлёта и прилёта
    type string Тип документа
    transport_type string Тип транспортного средства
    ticket_level string Класс обслуживания
    need_to_check boolean true — обязательно нужно перепроверить, false — система уверена в распознанных данных более чем на 90%
